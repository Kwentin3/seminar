# PRD-002: Cabinet Materials Auth v0.1

## Purpose / Scope
Этот draft PRD описывает первый продуктовый инкремент внутренней части сайта с авторизацией. Цель не в создании «кабинета вообще», а в запуске минимально полезной internal materials area, где команда сможет безопасно находить и открывать seminar-материалы.

Статус документа:
1. Это draft, обновлённый после first implementation slice.
2. В документе отражены уже принятые продуктовые решения v1.
3. Follow-up вопросы оставлены только там, где они не блокировали текущий срез.

## Context
Подтверждённые факты discovery:
1. Phase 1 уже дала лендинг, сбор лидов и read-only `/admin`.
2. Полноценной auth-системы, user model и session model сейчас нет.
3. Seminar materials уже существуют, но как repo-hosted markdown/PDF/doc assets, а не как управляемая библиотека.

Рабочая product recommendation для first slice:
1. Стартовать с `internal team cabinet`, а не с client cabinet.
2. Дать команде быстрый доступ к материалам и базовому search/filter.
3. Оставить client/speaker access как следующий продуктовый слой.

## Who Is The User
### Confirmed
1. В проекте уже есть внутренняя команда, которая работает с seminar-материалами.
2. В репозитории нет подтверждённой модели внешних пользователей кабинета.

### Target users for first increment
1. `admin`:
   - управляет users/roles later;
   - видит весь cabinet surface.
2. `viewer`:
   - входит в кабинет;
   - просматривает и открывает материалы.

### Explicitly deferred users
1. `editor`
2. `speaker`
3. `seminar-client`

Они не исключены навсегда, но не входят в first slice по умолчанию.

## User Stories
1. Как участник внутренней команды, я хочу войти в защищённую внутреннюю часть сайта, чтобы видеть seminar-материалы вне публичного лендинга.
2. Как участник внутренней команды, я хочу видеть список материалов с понятными названиями, типами и кратким описанием, чтобы быстро находить нужное.
3. Как участник внутренней команды, я хочу фильтровать материалы по теме, типу и аудитории, чтобы не искать их вручную по репозиторию.
4. Как участник внутренней команды, я хочу открывать markdown/PDF/external links из одного места, чтобы не прыгать между папками и документами.
5. Как администратор, я хочу иметь отдельный управляемый вход в cabinet без использования общего `ADMIN_SECRET`, чтобы доступ был персонализированным и отзывным.
6. Как лектор, я хочу читать markdown-материалы прямо внутри cabinet, чтобы использовать портал как рабочую опору при подготовке к семинару.
7. Как лектор, я хочу сразу видеть статус материала и curator-signals, чтобы понимать, это черновик, рабочий документ или уже опорный материал.

## Scope Of First Increment
1. Новый маршрут `"/cabinet"` внутри текущего приложения.
2. Login screen с username/email + password.
3. HttpOnly cookie-backed session после логина.
4. Роль `viewer` и роль `admin`.
5. Library view материалов:
   - список;
   - поиск по title/summary;
   - фильтры по type/topic/audience/language;
   - переход к материалу или открытие внешней ссылки.
6. Detail/read view для markdown-материалов внутри cabinet.
7. Начальный каталог материалов как curated metadata layer поверх уже существующих файлов и docs.
8. Сосуществование с legacy `/admin`, без немедленной замены.
9. Лёгкая status model для lecturer-oriented ориентации в библиотеке.

## Out Of Scope
1. Client-facing кабинет после семинара.
2. Magic link auth.
3. Social login / third-party identity provider.
4. Recommendation engine.
5. Отметка прогресса, completion tracking, learning analytics.
6. Полноценное редактирование материалов в UI.
7. Fine-grained permissions beyond `admin|viewer`.

## Functional Requirements
1. `FR-CAB-01`: Система должна предоставлять отдельный cabinet route, не смешанный с публичным lead flow.
2. `FR-CAB-02`: Неавторизованный пользователь не должен видеть cabinet content.
3. `FR-CAB-03`: Авторизованный `viewer` должен видеть только materials library surface.
4. `FR-CAB-04`: Авторизованный `admin` должен иметь тот же viewer access и задел под дальнейший admin-only cabinet surface.
5. `FR-CAB-05`: На первом экране после логина должен быть materials library, а не пустой dashboard.
6. `FR-CAB-06`: Каждая карточка материала должна содержать как минимум `title`, `material_type`, `summary`, `topic`, `audience`, `language`, `source_location`.
7. `FR-CAB-07`: Пользователь должен иметь возможность открыть локальный файл, markdown-doc или внешний URL из карточки материала.
8. `FR-CAB-08`: Поиск должен работать по `title`, `summary`, `tags/topic`.
9. `FR-CAB-09`: Фильтры должны работать минимум по `material_type`, `audience`, `language`.
10. `FR-CAB-10`: Legacy `/admin` и связанные admin endpoints должны продолжать работать до отдельного решения о замене.
11. `FR-CAB-11`: Markdown-материалы должны иметь in-app read route с frontmatter-free readable rendering.
12. `FR-CAB-12`: Cabinet UI должен различать материалы, которые читаются в портале, и материалы, которые открываются во внешнем просмотре.
13. `FR-CAB-13`: Каждый материал должен иметь человекопонятный readiness status из минимального набора `draft|working|final`.
14. `FR-CAB-14`: В list view и detail view должны отображаться lecturer-useful curation signals: status, theme, reading mode и базовый updated signal.
15. `FR-CAB-15`: Статусный слой должен оставаться curated metadata layer, а не workflow approval platform.

## Non-Functional Requirements
1. Stack compatibility:
   - Node.js backend;
   - SQLite;
   - без managed auth service по умолчанию.
2. Security baseline:
   - пароли только как salted hash;
   - session cookie `HttpOnly`;
   - explicit logout;
   - защита от brute-force хотя бы на уровне rate limit per IP/login attempt;
   - отсутствие секретов и паролей в логах.
3. UX baseline:
   - desktop-first with acceptable mobile usability;
   - cabinet usable without horizontal scrolling on `>=360px`.
4. Delivery baseline:
   - auth smoke tests обязательны;
   - regression для existing lead flow и `/admin` недопустим.

### Deferred roles

| `seminar-client` | deferred |
| `superadmin` | not needed for first slice |

## Materials Model
### Confirmed current state
1. Материалы уже лежат в repo как:
   - `docs/seminar/**/*.md`
   - `content/*.pdf`
   - `content/landing/*.json`
2. В БД материалов сейчас нет.

### Draft first-slice model
1. Хранить материалы как metadata records в SQLite.
2. Source of truth for file bodies may remain mixed:
   - repo markdown;
   - local VPS files;
   - explicit external URLs.
3. Material record minimum fields:
   - `id`
   - `slug`
   - `title`
   - `summary`
   - `material_status`
   - `material_type`
   - `topic`
   - `audience`
   - `language`
   - `theme`
   - `source_updated_at`
   - `recommended_for_lecture_prep`
   - `source_kind`
   - `source_path_or_url`
   - `is_active`
   - `created_at`
   - `updated_at`

## Acceptance Criteria / DoD
1. Cabinet route exists and is protected by user auth, not shared admin secret.
2. First admin can be bootstrapped without manual DB editing.
3. Authenticated `viewer` can browse and open curated materials.
4. Existing `/admin` and lead flow still pass smoke checks.
5. Materials inventory is externalized from ad hoc file browsing into explicit metadata records.
6. Auth/session behavior is covered by implementation tests and smoke scenarios.
7. Open questions below are either resolved or explicitly deferred before implementation start.
