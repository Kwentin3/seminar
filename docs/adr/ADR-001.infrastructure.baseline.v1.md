# ADR-001: Infrastructure Baseline (Phase 1)

## 1) Context
- Фаза 1: `Landing + Lead Engine + Minimal Admin`.
- Нужно зафиксировать базовые инфраструктурные решения по storage, защите `/admin`, anti-bot, нормализации телефона и performance baseline.
- Цель: минимализм, скорость внедрения и управляемость без платформенной перегрузки.

## 2) Decision
### 2.1 Storage
- Используется `Cloudflare D1`.
- Обоснование: табличная структура лидов, удобная сортировка/фильтрация, достаточная масштабируемость для горизонта 12-18 месяцев.

### 2.2 Admin Protection
- Для `/admin` используется header token gate через env secret (`ADMIN_SECRET`).
- Полноценная auth-система в Phase 1 не внедряется.
- `ADMIN_SECRET` хранится только в env и не размещается в коде/репозитории.
- Ротация `ADMIN_SECRET` выполняется вручную по регламенту (минимум раз в 30 дней).

### 2.3 Anti-Bot
- Для формы лида используется `Cloudflare Turnstile`.
- Выбран как минимально фрикционный базовый барьер против ботов.

### 2.4 Phone Normalization
- Страна определяется автоматически на backend.
- Телефон хранится только в формате `E.164`.
- Нормализация выполняется на backend до записи в storage.
- Если номер не нормализован автоматически, API возвращает `country_required`, а frontend запрашивает явный выбор страны.

### 2.5 Performance Baseline
- `Lighthouse mobile >= 85`.
- `TTI < 3s` на средних мобильных устройствах.
- Контроль и минимизация bundle size как обязательное ограничение при изменениях UI.

## 3) Consequences
Плюсы:
- Минимальная инфраструктура для быстрого запуска.
- Низкая операционная сложность на Phase 1.
- Решения соответствуют горизонту проекта 12-18 месяцев.

Минусы:
- Нет полноценной модели аутентификации/ролей для admin.
- Нет сложной антифрод-логики beyond Turnstile.
- Ограниченная CRM-логика и аналитическая глубина на старте.
- Появляется операционный overhead на ручную ротацию `ADMIN_SECRET`.
- В части кейсов конверсия может падать из-за дополнительного шага выбора страны.
