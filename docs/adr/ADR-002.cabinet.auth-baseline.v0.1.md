# ADR-002: Cabinet Auth Baseline v0.1

## Context
Discovery подтвердил:
1. текущий `/admin` защищён shared `ADMIN_SECRET` header gate и не даёт user identity;
2. проект не имеет `users`, `roles`, `sessions` и materials library model;
3. cabinet/auth меняет системную границу вокруг identity, storage и internal routing, поэтому ADR обязателен по `docs/ARCHITECTURE/CONTEXT_GOVERNANCE.md`.

При этом first slice должен оставаться совместимым с текущим стеком:
1. Node.js backend;
2. SQLite;
3. monorepo-lite layout;
4. без обязательного внешнего auth provider.

## Decision
Для first implementation slice принять следующий baseline:

1. Cabinet реализуется внутри текущего приложения как отдельный route-space `"/cabinet"`, а не как отдельный сервис или поддомен.
2. Auth реализуется как app-native username/email + password login.
3. Session model: server-validated session with HttpOnly cookie; session records and users live in SQLite.
4. Role model first slice: `admin` and `viewer`.
5. Legacy `/admin` на `ADMIN_SECRET` сохраняется как parallel legacy surface до отдельного решения о замене.
6. First admin bootstrap идёт через env-assisted startup seed:
   - `CABINET_BOOTSTRAP_ADMIN=1`
   - `CABINET_BOOTSTRAP_USERNAME`
   - `CABINET_BOOTSTRAP_EMAIL`
   - `CABINET_BOOTSTRAP_PASSWORD`
   - existing admin reset is explicit and requires `CABINET_BOOTSTRAP_ALLOW_RESET=1`
7. Materials library наполняется через registry-driven scan поверх repo materials, а в SQLite хранится только curated metadata layer.

## Alternatives Considered
1. Оставить shared secret gate как cabinet auth.
   - Rejected because it does not create user identity, role revocation or session control.
2. Magic link only.
   - Deferred because it добавляет зависимость от email delivery и ops/secrets surface без явной необходимости для internal-first slice.
3. Reverse-proxy auth/basic auth.
   - Rejected because access model и future materials roles должны жить в приложении и SQLite, а не только на edge.

## Consequences
### Positive
1. Решение остаётся в текущем стеке и не требует managed services.
2. Появляется персонализированный доступ вместо общего секрета.
3. Cabinet можно развивать постепенно от internal-only к richer access model.
4. `/admin` не ломается немедленно и остаётся rollback/legacy path.
5. Bootstrap becomes safer for v1 ops because forgetting the flag no longer silently resets the admin password on each restart.

### Negative
1. Появляются новые таблицы, миграции и auth smoke tests.
2. Нужны password hashing, session expiration, logout и brute-force guardrails.
3. Придётся поддерживать два access surface параллельно, пока `/admin` не будет либо absorbed, либо explicitly deprecated.
