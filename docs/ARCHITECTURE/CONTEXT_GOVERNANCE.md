# Context Governance

## 1) Purpose / Scope
Правила управления контекстом проекта "Семинары": как меняются ключевые документы, когда обязателен ADR, где проходят границы эпиков, и как ограничивается рост документационного домена.

Принципы:
- Узкий контекст - основа масштабирования.
- Документы важнее обсуждений.
- История решений должна быть прослеживаема.
- Минимальный рост документационного домена.
- Модель изменения: B-Light (PRD можно менять напрямую, стратегические изменения требуют ADR).

## 2) Document Lifecycle
| Status | Кто переводит | Условия перехода |
|---|---|---|
| `draft` | `owner` документа | Документ создан, структура заполнена минимумом секций. |
| `review` | `owner` документа | Изменение готово к проверке, заполнены `related` и `core_snapshot` (если EPIC). |
| `stable` | Утверждающий из `approved_by` | Есть минимум 1 `approved_by`, и этот handle не равен `owner`; для документов ядра минимум 2 `approved_by`. |
| `deprecated` | `owner` + утверждающий из `approved_by` | В документе есть ссылка на заменяющий документ в `related`. |
| `archived` | Утверждающий из `approved_by` для архитектурного домена | Документ не нужен в активной работе, но сохранен для трассировки истории. |

Ограничение: переход `draft -> stable` без `review` запрещен.

Механизм фиксации статуса:
1. Любой переход статуса делается отдельным PR с обновлением `status` и `last_updated`.
2. Переход в `stable/deprecated/archived` запрещен при пустом `approved_by`.
3. `owner` и `approved_by` всегда задаются github-handle, не названием роли.

## 3) Core Context Definition
В ядро контекста входят только следующие документы:

1. `docs/DOCS_CANON.md` - правила документационного домена.
2. `docs/prd/PRD.md` - источник продуктовых требований (planned, пока отсутствует в репозитории).
3. `docs/ARCHITECTURE/NORTH_STAR.md` - стратегический вектор и инварианты.
4. `docs/ARCHITECTURE/CONTEXT_GOVERNANCE.md` - данный регламент изменений.

Любой документ вне списка ядром не является.

## 4) B-Light Change Model
PRD можно менять напрямую, если изменение:
- не меняет архитектурные границы слоев/модулей;
- не создает новый домен, сервис или кросс-доменный контракт;
- не ломает действующие критерии качества/безопасности/эксплуатации.

ADR обязателен, если изменение:
- стратегическое или необратимое;
- влияет на несколько эпиков/доменов;
- меняет инварианты ядра контекста.

Связь ADR и PRD:
- PRD содержит ссылку на ADR в секции `related`.
- ADR содержит ссылку на PRD и пометку, какой пункт PRD изменен.

Пример:
- Сценарий "добавить мягкую анимацию на лендинг": правка PRD напрямую.
- Сценарий "вынести LLM-оркестрацию в отдельный сервис": сначала ADR, затем обновление PRD по принятому решению.

## 5) ADR Trigger Rules
ADR обязателен при любом из критериев:

1. Меняется граница между frontend/backend/functions.
2. Появляется новый внешний контракт (API, интеграция, формат обмена).
3. Меняется стратегия хранения/доступа к данным.
4. Меняются правила мультиязычности/доменного инварианта (`https://seminar-ai.ru/`).
5. Решение влияет более чем на один эпик или более чем на один домен.
6. Изменение трудно откатить без миграции или простоя.

## 6) Epic Boundaries
Правила:

1. Эпик не имеет права менять ядро контекста без ADR.
2. Эпик меняет только собственный домен (`landing`, `cabinet`, `ai`, `ops` и т.д.).
3. Кросс-доменные изменения оформляются как отдельный ADR + отдельный координационный эпик.
4. Эпик не стартует без `CONTEXT_SNAPSHOT.md`.

Эскалация изменения ядра:
1. Автор эпика фиксирует риск в EPIC-документе.
2. Создается ADR с пометкой `related` на эпик и ядро.
3. После принятия ADR обновляются ядровые документы.

## 7) Change Protocol
1. Инициировать: зафиксировать цель изменения и затрагиваемые документы.
2. Зафиксировать snapshot: для EPIC создать `docs/epics/EPIC-<index>.<domain>.<slug>/CONTEXT_SNAPSHOT.md`.
3. Оценить: определить класс изменения (`direct-PRD` или `ADR-required`) по разделам 4 и 5.
4. ADR (если нужно): создать ADR, провести review, перевести в `stable`.
5. Обновить PRD: внести только изменения, разрешенные принятым ADR или direct-PRD правилом.
6. Выполнить drift-check: после EPIC снова пройти ADR Trigger Rules.
7. Если drift-check сработал, а ADR нет: создать ADR и держать EPIC/REPORT в `review`.
8. Обновить `related`: двусторонние ссылки между PRD, ADR, EPIC и governance-документами.

### 7.1) Context Snapshot Rule
Перед началом каждого EPIC обязателен снимок контекста:

1. Файл: `docs/epics/EPIC-<index>.<domain>.<slug>/CONTEXT_SNAPSHOT.md`.
2. Содержимое: список документов ядра из раздела 3 с фиксированной датой и commit hash.
3. Все документы эпика ссылаются на снимок через поле `core_snapshot`.

Пример:
- `docs/epics/EPIC-001.landing.detox-ui/CONTEXT_SNAPSHOT.md`

### 7.2) Drift Detection Rule
После завершения каждого EPIC обязателен drift-check:

1. Сравнить фактические изменения с критериями раздела 5 (ADR Trigger Rules).
2. Если найдено хотя бы одно совпадение и ADR отсутствует - открыть ADR немедленно.
3. До появления ADR статус EPIC и итогового REPORT не может быть `stable`.

## 8) Minimalism Rule
Ограничения для ADR:

1. Не более 1 страницы текста.
2. Без пересказа PRD.
3. Только секции `Context`, `Decision`, `Consequences`.

Пример короткого ADR-скелета:
- `Context`: "Нужно разделить лендинг и кабинет по разным pipeline."
- `Decision`: "Вводим отдельные deployment jobs."
- `Consequences`: "Больше конфигурации CI, но меньше связности релизов."
