# PRD Phase 1: Landing

## 1) Purpose
Фаза 1 создает рабочий вход в B2B-воронку: лендинг, сбор лидов и минимальная админка для просмотра заявок. Это практический старт продаж через семинары, а не построение платформы.

Продукт этой фазы должен быстро запускаться, быть понятным с мобильного устройства и давать команде измеримый поток заявок. Любая функция, которая не улучшает качество лида или конверсию формы, в фазу не входит.

## 2) Scope
В фазу входит:

- Landing:
  - 3 роли аудитории:
    - `Business Owner` (владелец бизнеса / директор).
    - `Operations Lead` (руководитель операционного блока/подразделения).
    - `IT Lead` (руководитель IT/цифровой трансформации).
  - 3 user stories на роль:
    - `Business Owner`:
      - Как владелец бизнеса, я хочу быстро понять бизнес-эффект семинара, чтобы оценить окупаемость.
      - Как владелец бизнеса, я хочу увидеть реальные кейсы ошибок и результатов, чтобы снизить риск решения.
      - Как владелец бизнеса, я хочу оставить контакт за <1 минуту, чтобы быстро перейти к разговору.
    - `Operations Lead`:
      - Как операционный руководитель, я хочу увидеть типовые сценарии внедрения, чтобы соотнести их с моей болью.
      - Как операционный руководитель, я хочу понять формат семинара и ожидаемый результат, чтобы обосновать участие команде.
      - Как операционный руководитель, я хочу оставить телефон без сложной регистрации, чтобы не терять время.
    - `IT Lead`:
      - Как IT-руководитель, я хочу видеть практический подход и ограничения, чтобы оценить применимость.
      - Как IT-руководитель, я хочу понимать, что это не "магия", а методология внедрения, чтобы согласовать реалистичный план.
      - Как IT-руководитель, я хочу связаться через форму и получить follow-up, чтобы обсудить технические рамки.
- RU/EN i18n:
  - Переключение языка через UI.
  - Все ключевые блоки и форма доступны на RU и EN.
- Light/Dark theme:
  - Переключение темы через UI.
  - Состояние темы сохраняется между сессиями.
- Lead capture form:
  - Поля: `name`, `phone`.
  - Валидация на клиенте и сервере.
- Backend API endpoint:
  - Endpoint приема лидов на Node.js backend (`/api/leads`).
- Lead storage (минимальное):
  - Персистентное хранение лидов в минимальной структуре.
- Admin route:
  - Маршрут `/admin` для просмотра списка лидов.
  - Минимальная защита через секрет в env (без user cabinet).
- CI/CD + deployments:
  - CI проверки на pull request.
  - Production deploy на VPS по runbook.

## 3) Out of Scope
- SMS auth.
- User cabinet.
- LLM analytics.
- Complex CRM.
- Full integrations (CRM/ERP/messaging).

## 4) Functional Requirements
### 4.1 UI behavior
- `FR-UI-01`: Лендинг имеет минимум следующие блоки: value proposition, user stories, CTA с формой.
- `FR-UI-02`: Язык (`RU/EN`) переключается без перезагрузки страницы.
- `FR-UI-03`: Тема (`Light/Dark`) переключается без перезагрузки страницы.
- `FR-UI-04`: Форма отправляется с двумя полями (`name`, `phone`), кнопка недоступна при невалидных данных.
- `FR-UI-05`: После успешной отправки показывается подтверждение и очищение формы.
- `FR-UI-06`: При ошибке API показывается понятное сообщение и возможность повторить отправку.
- `FR-UI-07`: Если номер не удалось валидировать/нормализовать автоматически, форма запрашивает явный выбор страны и повторную отправку.

### 4.2 API behavior
- `FR-API-01`: `POST /api/leads` принимает JSON `{ name, phone, locale, source }`.
- `FR-API-02`: Сервер валидирует обязательность `name` и `phone`, нормализует `phone`.
- `FR-API-03`: На успех возвращается `200` + `lead_id`.
- `FR-API-04`: На невалидные данные возвращается `400` с кодом ошибки.
- `FR-API-05`: На внутреннюю ошибку возвращается `500` без утечки чувствительных деталей.
- `FR-API-06`: Дубликаты по одинаковому `phone` в окне 24 часов блокируются с `409 duplicate_lead`.
- `FR-API-07`: Если автоопределение страны не дало корректной нормализации, API возвращает `422` и код `country_required`.

### 4.3 Admin behavior
- `FR-ADM-01`: `/admin` показывает таблицу лидов (дата, имя, телефон, страна, язык, источник).
- `FR-ADM-02`: Доступ к `/admin` возможен только при корректном секретном токене/пароле из env.
- `FR-ADM-03`: Админка read-only в Фазе 1 (без редактирования/удаления).
- `FR-ADM-04`: Сортировка по дате (новые сверху) доступна по умолчанию.

## 5) Non-Functional Requirements
- Mobile-first:
  - Основной сценарий (просмотр + отправка формы) полноценно работает на экранах от `360px`.
- Performance baseline:
  - Lighthouse Performance (mobile, preview): `>= 85`.
  - TTI на главной: `< 3s` на среднем мобильном профиле.
  - LCP на главной: `<= 2.8s` на типовом мобильном профиле.
  - CLS: `<= 0.1`.
- Clean minimal UX:
  - Не более одного primary CTA на первом экране.
  - Нет блокирующих попапов до взаимодействия пользователя.
- Basic security principles:
  - Серверная валидация и санация входных данных.
  - Секреты только в env, не в клиентском бандле.
  - `/admin` не доступен без секрета.
  - `ADMIN_SECRET` ротируется вручную не реже 1 раза в 30 дней.
  - Логи не содержат полный phone в открытом виде.

## 6) Technical Constraints
- Hosting/runtime: JustHost VPS.
- Server runtime: Node.js service (`server/index.mjs`) под `systemd` + `nginx` reverse proxy.
- Repo layout: monorepo-lite (frontend + backend в одном репозитории, без избыточной сервисной декомпозиции).
- Env handling:
  - Локальные и production env разделены.
  - Обязательные переменные:
    - `ADMIN_SECRET`
    - `DATABASE_PATH`
  - Опциональные переменные:
    - `TURNSTILE_SECRET_KEY`
  - `ADMIN_SECRET` хранится только в env и не коммитится в репозиторий.
  - Ротация `ADMIN_SECRET` выполняется вручную по регламенту (минимум раз в 30 дней).
  - Отсутствие обязательной env должно падать при старте функции с явной ошибкой.

## 7) Definition of Done
Фаза 1 считается завершенной, если выполнены все критерии:

1. Лендинг опубликован на production и доступен в RU/EN.
2. Переключение Light/Dark работает и сохраняет состояние.
3. Форма `name + phone` отправляет лид в API и показывает корректный success/error UX.
4. `POST /api/leads` проходит позитивные и негативные тест-кейсы (валидные, невалидные, пустые поля).
5. Лиды сохраняются персистентно и видны в `/admin`.
6. `/admin` защищен секретом из env и не открывается без авторизации.
7. CI для pull request проходит green (lint/typecheck/build/smoke).
8. Production деплой на VPS проходит по runbook и проверен smoke-check.
9. NFR-базовые метрики достигнуты (`Lighthouse >= 85`, `TTI < 3s`, `LCP <= 2.8s`, `CLS <= 0.1`).
10. Документация обновлена: связи в `related` и краткий отчет по запуску в `docs/reports/YYYY-MM-DD/`.
