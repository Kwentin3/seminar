# LLM Office Work Knowledge Domain Map

## Purpose / Scope
Этот документ фиксирует стартовую карту домена знаний по теме использования LLM в офисной работе. Его задача: дать устойчивую рамку для дальнейших research-документов, образовательных материалов и методологических артефактов без расползания контекста.

Это:
- не PRD продукта;
- не рыночный обзор;
- не методичка семинара;
- не библиотека промптов;
- не технический проект AI-платформы;
- а корневой доменный каркас для последующей декомпозиции.

Фокус документа:
- негативный пользовательский опыт при работе с LLM в офисе;
- причины системных провалов;
- базовые концепции успешного внедрения;
- structured outputs, adaptive prompting, AI literacy, workflow-based usage;
- границы домена и следующая очередь документов.

## Context
Документ создается внутри seminar knowledge-домена проекта "Семинары" и должен соответствовать трем ограничениям:

1. `docs/ARCHITECTURE/NORTH_STAR.md` задает проект как B2B-вход в корпоративные сделки через образовательные продукты, а не как SaaS-платформу.
2. `docs/ARCHITECTURE/CONTEXT_GOVERNANCE.md` требует минимального роста документационного домена и явных границ для новых knowledge-веток.
3. `docs/seminar/INDEX.md` фиксирует seminar-домен как отдельную ветку документации и не допускает смешения с landing- и shared core-материалами.

Отсюда следуют рабочие рамки:
- этот документ описывает предметную область для семинаров, диагностики и методологии внедрения;
- документ не уходит в productization отдельной AI-платформы;
- центральная проблема формулируется не как "какая модель лучше", а как разрыв между chat-first интерфейсом и реальной офисной работой;
- домен должен допускать рост через дочерние research-документы, а не через разрастание одного текста.

## Main Section

### 1. Domain Positioning
Основной тезис домена: негативный опыт использования LLM в офисе возникает не только из качества модели, а из системного разрыва между:
- chat-first взаимодействием;
- реальной офисной работой;
- шаблонными документами;
- структурированными данными;
- повторяемыми бизнес-процессами;
- требованиями доверия, качества и безопасности.

В рамках этого домена LLM рассматривается прежде всего как инструмент работы с контекстом и промежуточными представлениями, а не как "магический чат", который должен сразу выдавать готовый корпоративный документ.

### 2. Problem Frame
Высокоуровневая рамка проблемы:

1. Пользователь обычно воспринимает LLM как чат-бота, а не как систему контекстной сборки результата.
2. Пользователь часто сразу требует финальный артефакт вместо поэтапной структуры, данных и черновика.
3. В офисной среде постоянно конфликтуют смысл, форма, шаблон и регламент.
4. LLM надежнее работает как генератор черновика, структуры и данных, чем как финальный редактор корпоративного документа.
5. Chat-first подход удобен для первого входа, но слаб для повторяемой офисной работы.
6. Плохой результат часто порождается не слабостью модели, а плохой постановкой задачи и дефицитом контекста.

Негативный пользовательский опыт обычно выглядит так:
- пользователь задает вопрос вместо постановки задачи;
- просит готовый документ без исходных данных и ограничений;
- получает текст, который не совпадает с корпоративным шаблоном;
- вынужден вручную копировать, вставлять и исправлять;
- перестает доверять LLM как рабочему инструменту.

### 3. Foundational Concepts
Базовые концепции домена:

| Концепт | Короткая фиксация |
|---|---|
| LLM | Вероятностный генератор, а не "знающий эксперт". |
| Контекст | Ключевой ресурс качества: чем лучше собран контекст, тем выше полезность выхода. |
| Вопрос vs задача | Вопрос инициирует диалог, задача задает целевую работу, ограничения и критерии качества. |
| Текст vs данные vs документ | Текст - свободная языковая форма; данные - структурированное представление; документ - оформленный артефакт в шаблоне и регламенте. |
| Черновик vs structured output vs финальный артефакт | Черновик помогает думать, structured output помогает валидировать и интегрировать, финальный артефакт требует редакторской и процессной сборки. |
| Документ как форма | Документ - это не истина сама по себе, а способ представления смысла в корпоративной среде. |
| Данные отдельно от формы | Для надежности смысл и структура должны жить отдельно от окончательного форматирования. |

Практическое следствие:
- офисной аудитории нужно объяснять ценность `JSON`, `CSV`, `XML` и других структурированных представлений;
- structured outputs легче валидировать;
- их легче встроить в CRM, ERP, 1C, BI, шаблоны документов и повторяемые процессы;
- шаблон должен оформлять, а LLM должна наполнять смыслом и структурой.

### 4. Problem Domains
Проблемное поле домена разбивается на следующие кластеры:

| Кластер | Содержание |
|---|---|
| Ошибочная ментальная модель пользователя | LLM воспринимается как чат-ассистент общего назначения, а не как система работы с контекстом, структурой и ограничениями. |
| Отсутствие навыка постановки задачи | Пользователь не разделяет цель, входные данные, ограничения, формат выхода и критерии качества. |
| Конфликт LLM с офисными форматами и шаблонами | Свободный текст плохо совпадает с корпоративными формами, реквизитами, табличными шаблонами и утвержденными структурами. |
| Галлюцинации и кризис доверия | Нестабильность фактов и формулировок быстро приводит к репутации "нельзя использовать в работе". |
| Отсутствие контроля качества | Нет явной валидации, чек-листов, ответственного review и human-in-the-loop процедуры. |
| Отсутствие интеграции в реальный workflow | Результат нельзя встроить в повторяемый процесс, поэтому он остается одноразовым чат-ответом. |
| Безопасность и чувствительные данные | Пользователь не понимает red lines, режимы безопасного использования и риски утечки контента. |
| Разрыв между power users и обычными офисными пользователями | Лучшие практики остаются у немногих продвинутых сотрудников и не превращаются в массовый стандарт. |

### 5. Solution Domains
Крупные направления решений:

| Направление | Назначение |
|---|---|
| Prompt literacy | Обучение постановке задач, контексту, ограничениям и формату результата. |
| Self-prompting | Использование LLM для совместной сборки более точного промпта и структуры задачи. |
| Adaptive prompting | Динамические шаблоны, которые меняются под тип задачи, роль пользователя и доступный контекст. |
| Prompt libraries | Базовые заготовки для повторяемых сценариев, если они не оторваны от workflow и не статичны навсегда. |
| Structured outputs | Генерация `JSON`/`CSV`/`XML`/табличных структур вместо свободного текста там, где нужна встраиваемость и проверяемость. |
| Workflow-based usage | Проектирование повторяемого контура: сбор контекста -> структура -> данные -> валидация -> сборка артефакта. |
| AI literacy | Объяснение ограничений модели, типов ошибок и реалистичных режимов применения. |
| Data literacy | Обучение тому, как данные и схемы поддерживают качество, валидацию и интеграцию. |
| Governance / safe usage / red lines | Правила обращения с чувствительным контентом, допустимыми сценариями и ответственностью человека. |
| AI champions | Внутренние носители практики, которые превращают единичные находки в повторяемые рабочие подходы. |

Ключевая мысль этого слоя: компании покупают не "доступ к модели", а снижение хаоса, рост повторяемости и более надежное встраивание AI в процессы.

### 6. Workflow Model
Предпочтительная operational-модель для офисного применения:

1. Сбор контекста.
2. Формирование структуры задачи.
3. Генерация данных или смысловых блоков.
4. Валидация по схеме, чек-листу или человеку.
5. Сборка финального артефакта через шаблон, форму или корпоративный инструмент.

Этот контур важен потому, что:
- повторяемая офисная работа требует процессов, а не только чатов;
- документ является последним слоем представления, а не единственным объектом генерации;
- LLM надежнее использовать для генерации структуры и данных, чем для окончательного оформления.

### 7. Research Clusters
Направления следующей декомпозиции:

1. Negative user experience in office LLM usage.
2. Enterprise adoption barriers.
3. Office workflow integration.
4. Adaptive prompt systems.
5. Structured outputs and schema-guided generation.
6. AI literacy for non-technical users.
7. Document workflows vs data workflows.
8. Practical teaching methods for corporate seminars.
9. Prompt libraries and repeatable scenarios.
10. Quality control and human-in-the-loop.

### 8. Domain Boundaries
Пока не входят в основной фокус этого домена:

1. Глубокая агентная архитектура.
2. RAG как центральная тема.
3. Векторные базы как самостоятельный фокус.
4. Полная техническая реализация AI agents.
5. Low-level API-specific prompting.
6. Deep software architecture of AI platforms.
7. Детальный vendor comparison моделей и платформ.

Причина ограничений: текущий домен нужен для методологии использования LLM в офисной работе и корпоративном обучении, а не для построения отдельного engineering-направления вокруг AI-инфраструктуры.

### 9. Terminology Starter Set
Стартовый словарь домена:

| Термин | Определение |
|---|---|
| Chat-first | Подход, в котором взаимодействие с LLM строится как свободный диалог без явной процессной структуры. |
| Prompt literacy | Навык формулировать задачу, контекст, ограничения и ожидаемый формат выхода. |
| AI literacy | Базовое понимание того, что умеет и чего не умеет LLM, где ее ограничения и риски. |
| Data literacy | Понимание роли структурированных данных, схем и табличных форм в рабочем процессе. |
| Structured output | Вывод в заранее определенной структуре, пригодной для валидации и интеграции. |
| Workflow | Повторяемая последовательность шагов, в которую можно встроить LLM. |
| Context engineering | Целенаправленная сборка и подача релевантного контекста под конкретную задачу. |
| Self-prompting | Практика, в которой LLM помогает пользователю собрать или улучшить промпт. |
| Adaptive prompt | Промпт-шаблон, который меняется в зависимости от сценария, данных и цели. |
| Prompt library | Набор переиспользуемых шаблонов для типовых задач. |
| Human-in-the-loop | Режим, в котором человек выполняет review, валидацию или финальное утверждение результата. |
| Hallucination | Правдоподобный, но недостоверный или выдуманный вывод модели. |
| Office automation | Автоматизация повторяемых офисных операций, где AI является частью процесса, а не изолированным чатом. |
| Document workflow | Процесс, ориентированный на создание, согласование и выпуск документов. |
| Data workflow | Процесс, ориентированный на сбор, преобразование, проверку и передачу данных между системами. |
| Schema-guided generation | Генерация по заранее заданной схеме или структуре полей. |
| Draft | Черновой результат, полезный для дальнейшей доработки, но не равный финальному артефакту. |
| Final artifact | Итоговый корпоративный объект: документ, письмо, отчет, карточка, форма или запись в системе. |
| Template | Нормированная форма представления результата в корпоративной среде. |
| Validation | Проверка качества, полноты, формата и допустимости результата. |
| Safe usage | Набор правил, ограничивающих рискованное использование AI в рабочих сценариях. |
| Red lines | Явные запреты на обработку определенных данных, задач или режимов использования. |
| Trust gap | Разрыв между ожиданиями пользователя и фактической надежностью результата, ведущий к недоверию. |
| Power user | Сотрудник, который умеет получать от LLM устойчиво более полезные результаты, чем массовый пользователь. |
| Knowledge domain map | Каркасный документ, который задает состав домена, его границы и направления декомпозиции. |

1. Он работает как карта домена, а не как эссе, PRD или рыночный обзор.
